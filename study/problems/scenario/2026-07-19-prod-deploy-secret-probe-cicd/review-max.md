## 문제 1. 인프라 자원 사전 준비 (22회차)

### (a) 3306 포트 · Public Access 허용 이유 — 부분 충족

**제출 요지**
> "ec2 안에 떠있는 pod가 db에 연결되려면 3306 포트를 열어줘야함. public access를 허용한 이유는 외부에서
> 접근할 db툴을 연결해서 사용하기 위함."

**맞은 점**
- 3306 개방 이유(Pod→RDS 연결) 정확.
- Public Access 허용 목적(로컬 DB 툴 접속)도 정확 — 흔한 오답인 "EC2에서도 접근이 안 돼서"류 오해에
  빠지지 않음.

**놓친 점**
- 문제가 명시적으로 요구한 "같은 **AWS 내부 접근과 무엇이 다른지**"에 대한 대조 설명이 없다. 즉
  "AWS 내부(EC2 등)는 인바운드 포트만 열면 이미 접근 가능하지만, 로컬 PC는 AWS 바깥이라 별도로
  Public Access를 열어야 한다"는 대조 논리가 빠져 결론만 맞고 근거의 절반이 비어 있다.

### (b) Redis Ingress 미사용 · Service 경유 이유 — 충족

**제출 요지**
> "레디스는 외부에서 접근할 필요가 없기때문에... pod는 언제든 다시 생성되거나 할수 있기때문에 ip가
> 변경되게 되므로 안정적인 서비스 이름으로 서비스를 통해 통신하기 위함."

**맞은 점**
- Ingress 미사용 이유(내부 전용 접근), Service 경유 이유(Pod IP 가변성 vs 안정적 이름) 모두 루브릭과
  정확히 일치.

### (c) 인프라 생성 vs CI/CD, 반복/일회성 — 충족

**제출 요지**
> "RDS/Redis/ECR을 만드는 작업들은 한번만 작업해두면 되는 일회성 작업... GitHub Actions CI/CD는 코드가
> 계속 변경되고 자주 배포되면서 반복되는 작업."

**맞은 점**
- 루브릭의 핵심(빈도 차이에 근거한 자동화 대상 구분)과 정확히 일치.

## 문제 2. 이미지 빌드와 민감정보 분리 판단 (23회차)

### (a) profile 값과 on-profile 매칭 — 부분 충족

**제출 요지**
> "application.yaml은 공통 설정이기 때문에 스프링부트가 먼저 로드함. 따로 프로필이 지정되어 있지 않으면
> profile.default 값과 매칭되는 yaml 파일이 활성화된다."

**맞은 점**
- application.yaml이 공통 설정으로 먼저 로드된다는 점, "매칭"이라는 핵심 동작 개념은 짚음.

**틀린/놓친 점**
- 질문의 핵심은 "**profile 값**과 **각 프로필 전용 파일의 `config.activate.on-profile` 값**이 일치해야
  적용된다"는 일반 규칙인데, 제출 답변은 "프로필이 **지정되어 있지 않을 때**"의 예외 케이스만 다뤘다.
  `config.activate.on-profile` 필드 자체를 언급하지 않아, 명시적으로 프로필을 지정한 일반적인 경우
  (local/prod)의 매칭 메커니즘을 설명하지 못했다.

### (b) DB_HOST/DB_PW 이미지 미포함 이유 · 주입 시점 — 충족

**제출 요지**
> "민감한 정보가 외부에 노출되지 않도록... 실제 값은 secret에 따로 저장해두고있다. pod가 실행될때 yaml에
> 한번 주입된다."

**맞은 점**
- 이유(노출 방지)와 주입 시점(Pod 실행 시)이 루브릭과 일치. ("yaml에 주입"은 "env로 주입"의 다소 부정확한
  표현이나 본질은 맞음.)

### (c) DB 자격증명 vs JWT 시크릿화 판단 — 부분 충족

**제출 요지**
> "조작당해도 위협받을게 없기때문. 학습용이라 민감데이터가 있는것도 아님. 실사용자에게 오픈한
> 서비스라면 절대 노출되면 안됨."

**맞은 점**
- 위험 기반 판단이었다는 논리(피해 없음 → 시크릿화 안 함)를 정확히 짚음.
- 실사용자가 생기면 판단이 바뀌어야 한다는 결론도 맞음.

**놓친 점**
- 루브릭이 요구하는 **구체적 위험 기제**("JWT 위조 시 타인 계정 접근·권한 상승이 가능해져 전제가
  깨진다")를 설명하지 않고 "절대 노출되면 안 됨"이라는 결론만 재확인했다. 왜 위험한지(위조 가능성 →
  계정 탈취) 메커니즘이 빠져 있어, 판단 근거의 논리 사슬이 완성되지 않았다. (키 교체 시 기존 토큰
  무효화 등 가점 요소는 당연히 없음.)

## 문제 3. Secret 운영 실습과 자격증명 관리 (24회차 + 26회차)

### (a) Secret 생성 명령 · 원문 미노출 이유 — 부분 충족

**제출 요지**
> "인코딩되어 저장되기 때문."

**맞은 점**
- "암호화"가 아니라 "인코딩"이라 표현해 흔한 오답(암호화로 착각)은 피함.

**틀린/놓친 점**
- 문제가 명시적으로 "**명령을 쓰고**... 이유를 서술하라"고 두 가지를 요구했는데, 실제 `kubectl create
  secret generic myappsecret --from-literal=...` 명령 자체가 답변에 없다. 이유만 있고 실습 스킬(명령
  문법)을 검증할 근거가 빠졌다.
- "인코딩"이라고만 하고 구체적으로 **Base64**라는 점(따라서 디코딩으로 원문 복원 가능하다는 실무적
  함의)까지는 명시하지 않았다.

### (b) Secret 키 추가 절차 · env 이름 일치 대상 — 부분 충족

**제출 요지**
> "새로 키를 추가하고 싶다면 그냥 싹 지우고 키를 추가한다음 다시 생성한다. `kubectl delete secret
> secret-name -n namespace`, `kubectl create secret generic secret-name -n namespace
> --from-literal=key=value`... env에서 JWT_SECRET 값을 가져올때는 key가 JWT_SECRET이고 name과
> `${변수명}`을 같게 해서 코드상에서 불러온다."

**맞은 점**
- 삭제 후 재생성이라는 절차, `secretKeyRef`의 key와 `application.yaml`의 `${JWT_SECRET}` 변수명이
  일치해야 한다는 점을 정확히 짚음.

**놓친 점**
- "키를 추가한다음 다시 생성한다"는 표현이 **기존에 있던 DB_HOST/DB_PW 등 다른 키도 함께 포함해서
  재생성해야 한다**는 점을 명시적으로 언급하지 않았다. 실무에서 이 부분을 빠뜨리면 재생성 시 기존
  키가 통째로 사라져 실제 장애로 이어질 수 있는 지점이라 루브릭이 "필요한 키를 모두 포함해"라고 굳이
  강조한 대목인데, 이 뉘앙스가 빠졌다.

### (c) AWS 장기 키 위험 — 충족

**제출 요지**
> "시크릿에 넣어두고 사용하는건 기본적으로 만료기간이 길다는건데 한번 유출되면 위험이 커짐. 특히
> 만약 어드민 권한이 있으면 위험."

**맞은 점**
- 장기 자격증명 특성, 유출 시 만료 전까지 악용 가능, 어드민 권한 보유 시 계정 전체 위험 — 루브릭 핵심
  3가지 중 2가지(장기성/유출 시 악용, 어드민 권한 위험)를 정확히 짚음.

**놓친 점(경미)**
- 로테이션을 수동으로 관리해야 하는 부담은 언급되지 않았으나 핵심 논지는 충분히 성립.
- 범위 밖(OIDC)은 "모름"으로 명시 — 문제에 명시된 감점 없는 항목이라 문제 없음.

## 문제 4. 무중단 배포 안정성과 트러블슈팅 (24회차)

### (a) readinessProbe가 무중단 배포에 필수인 이유 — 부분 충족

**제출 요지**
> "pod는 running 중이더라도 어플리케이션이 동작하는 중이 아닐수 있기때문."

**맞은 점**
- Running ≠ 실제 서비스 가능 상태라는 전제는 정확.

**놓친 점**
- 루브릭이 요구하는 **인과 관계**("/health가 OK를 받아야 새 Pod가 준비됐다고 판단하고, 그제서야 기존
  Pod를 내리므로 스프링 기동 중(10~20초) 트래픽이 새 Pod로 가는 것을 막아준다")까지는 서술하지
  않았다. "왜 필요한가"의 전제만 있고, readinessProbe가 실제로 **어떻게** 다운타임을 막는지(기존
  Pod 유지 → 새 Pod 트래픽 차단) 메커니즘 설명이 빠졌다.

### (b) requests/limits 미설정 문제 · 점심 피크 대응 값 설정 — 부분 충족(약함)

**제출 요지**
> "pod 하나가 리소스를 다 잡아먹을 수 있음."

**맞은 점**
- 미설정 시 노이지 네이버 문제(한 Pod가 자원 독점)라는 전반부 답은 루브릭과 일치.

**틀린/놓친 점**
- 문제는 두 갈래 질문이다: ①미설정 문제, ②**점심 피크가 있는 이 서비스에서 requests/limits를 어떻게
  다르게 잡아야 하는지 근거와 함께**. 제출 답변은 ①만 한 줄로 답하고 ②는 아예 답하지 않았다.
  `limits`을 `requests`보다 여유 있게 둬서 평소엔 적게 예약하고 피크 때 버스트하도록 하는 설계
  판단(예: 0.5/250Mi ↔ 1/500Mi)이 완전히 빠져, 흔한 오답 목록의 "requests=limits로 답해 버스트
  여지를 고려하지 않음"보다도 더 낮은 완성도(값 언급 자체가 없음)다. 이 시나리오의 명시적 제약
  ("점심 피크 시간대 5xx 무발생")과 직결되는 항목이라 실무 임팩트가 크다.

### (c) revisionHistoryLimit 이유와 트레이드오프 — 충족

**제출 요지**
> "불필요한 오브젝트 누적 방지, 오래된 이력은 의미가 별로 없음. 가시성이 좋음. 트레이드오프는 롤백
> 범위 축소."

**맞은 점**
- 리비전 누적 방지(Argo CD 등 UI 혼란 감소와 같은 취지) + 트레이드오프(롤백 가능 범위 축소) 모두 정확.

### (d) 핸즈온: Running 0/1 트러블슈팅 — 부분 충족

**제출 요지**
> "kubectl get pod -n namespace, kubectl logs -f podname -n namespace 명령어로 빠르게 오류 확인.
> 만약 readinessProbe 설정이 없었다면 오류발생 상황에서 계속 트래픽을 받았을것이다."

**맞은 점**
- `get pods` → `logs -f`로 좁혀가는 큰 방향은 맞고, 로그로 원인을 특정한다는 접근 자체는 유효.

**틀린/놓친 점**
- 루브릭의 3단계(`get pods` → `describe pod`로 **Events 확인**[스케줄링 문제 의심 시] →
  `logs -f`[내부 에러 의심 시]) 중 `describe pod` 단계가 완전히 빠졌다. 스케줄링 실패(리소스 부족,
  이미지 pull 실패 등)처럼 로그 자체가 없는 경우엔 `describe`가 필수인데 이 경로가 없다.
  강의에서 실제로 겪은 예시(스키마 이름 오타 → `unknown database`)도 언급하지 않았다.
- readinessProbe 부재 시나리오 설명이 부정확하다. 제출은 "오류발생 상황에서 계속 트래픽을 받았을
  것"이라고 했는데, 루브릭 핵심은 **"DB 연결에 실패한(=미준비) 새 Pod가 Ready로 오인되어, 아직 정상
  동작 중인 기존 Pod가 먼저 내려가는" 다운타임 시나리오**다. 즉 "고장난 새 Pod가 트래픽을 받는다"는
  현상 설명에 가깝고, "그 대가로 정상이던 기존 Pod가 먼저 죽는다"는 무중단 배포의 핵심 위험까지는
  짚지 못했다 — 이는 answer-key의 "흔한 오답/누락 포인트"에 정확히 언급된 패턴이다.

## 문제 5. HTTPS 인증서 발급 (25회차) — 4개 항목 전부 미충족

**제출 요지 (a~d 공통)**
> "잘모름"

이 문제는 `problem.md`에 `⚠ 강의 범위 밖` 표시가 없는 **필수 문제**다(25회차 강의 범위 내 내용).
따라서 감점 없이 넘어갈 수 있는 항목이 아니라 전부 미충족으로 처리한다.

- (a) 대칭키 대신 비대칭키를 쓰는 구조적 이유 — 미충족.
- (b) 공개키/비밀키 생성 → CSR → CA 검증 → 인증서 발급 흐름 — 미충족.
- (c) `cert-manager`/`ClusterIssuer`/`Certificate` 역할 구분 — 미충족.
- (d) `Certificate` Ready=False 시 우선 의심 원인(Ingress http 라우팅 미비) — 미충족. 특히 이 항목은
  루브릭이 "흔한 오답"으로 별도 지목한 지점("CA 문제로만 답하고 Ingress 선행 조건을 놓침")인데,
  응답이 없어 그 함정 자체를 판단할 기회도 없었다.

## 문제 6. GitHub Actions CI/CD 파이프라인 설계 (26회차)

### (a) workflow/event/job/runner 구분 · 경로 규칙 — 충족

**제출 요지**
> "workflow는 자동화 프로세스를 정의하는 단위, event는 workflow를 실행하는 트리거, job은 실제
> 작업단위, runner는 실제로 job을 실행하는 실행 환경... workflows 폴더에 yaml 파일로 정의하도록
> 약속이 되어있기 때문."

**맞은 점**
- 4개 개념의 계층 구조와 경로 규칙의 이유(컨벤션) 모두 정확히 일치. 흔한 오답("다 같은 뜻으로 뭉뚱그림")
  회피.

### (b) 반복 vs 최초 1회 작업 구분 · 실제 반복 스텝 — 부분 충족

**제출 요지**
> "일회성 작업은 인프라 리소스 생성(eks, ecr, ec2 등등), 반복작업은 소스코드 변경으로 이미지
> 빌드/푸시, 파드 재생성. 실제로 workflow가 매번 반복하는 작업은 가상환경 생성, 필요한 프로그램
> 설치, ecr 로그인, 이미지 build/push, kubectl rollout restart로 pod 재생성."

**맞은 점**
- 반복/일회성 구분의 큰 축과, 실제 반복 스텝(가상환경 세팅 → 로그인 → 빌드/push → restart) 나열은
  루브릭과 대체로 일치. 흔한 오답("인프라 사전 생성까지 반복 작업에 포함")은 정확히 피함.

**놓친 점**
- 일회성 작업 목록이 "인프라 리소스 생성"에만 한정되어 있어, 루브릭이 명시한 **Secret/Service/Ingress
  최초 생성, cert-manager 자원 적용**까지는 포함하지 않았다(범위가 좁음).
- 반복 스텝에서 **체크아웃(git checkout)** 단계가 빠졌다. "가상환경 생성"(runner VM 세팅)과
  체크아웃(소스 코드 가져오기)은 서로 다른 스텝인데 이를 구분해 언급하지 않았다.

### (c) Deployment yaml 자주 바뀔 때 추가할 것 — 충족

**제출 요지**
> "kubectl apply -f deployment.yaml"

**맞은 점**
- 루브릭이 요구하는 정확한 답(매 실행마다 `kubectl apply -f <deployment 파일>` 추가)과 일치.
- 범위 밖(Blue-Green/Canary)은 "모름"으로 명시 — 문제에 감점 없음이 명시된 항목이라 문제 없음.

## 복습 포인트

1. **질문이 두 갈래일 때 뒷부분을 빠뜨리는 패턴**이 반복된다(문제2-a의 on-profile 매칭, 문제4-b의
   피크 대응 값 설정 근거, 문제4-d의 readinessProbe 부재 시 정확한 다운타임 메커니즘). 서술형 문제를
   풀 때 질문을 문장 단위로 쪼개 각 절이 답변에 대응하는지 스스로 체크하는 습관이 필요하다.
2. 문제4-b는 이 시나리오의 명시적 제약("점심 피크 5xx 무발생")과 직결되는 항목인데 값 설계
   근거가 전혀 없었다 — `requests`를 낮게, `limits`을 여유 있게 잡는 버스트 전략을 구체적 비율로
   설명할 수 있는지 다시 짚어볼 것.
3. 문제4-d에서 "readinessProbe가 없으면 무슨 일이 벌어지는가"는 "고장난 새 Pod가 트래픽을 받는다"가
   아니라 "정상이던 **기존** Pod가 먼저 내려간다"는 것이 핵심이다 — 롤링 업데이트에서 Ready 판정이
   신구 Pod 교체 타이밍에 어떻게 관여하는지 다시 확인해볼 것.
4. 문제 5(HTTPS/cert-manager)는 이번 제출에서 통째로 비어 있다 — 25회차 스크립트(대칭키/비대칭키,
   CSR/CA 검증 흐름, cert-manager/ClusterIssuer/Certificate 역할 분리, Ready=False 트러블슈팅)를
   별도로 복습할 것을 권장한다.
