# 채점 루브릭: 클러스터 구성 방식 결정 + 스케줄링 장애 대응

이 문제는 정답이 하나로 정해지지 않는다. 아래는 "반드시 언급해야 할 핵심 포인트"와
"모범 답안 예시"이며, 이 루브릭에 없어도 타당한 근거가 있으면 인정한다.

## 문제 1 — 환경 분리 (핵심 포인트)

- [ ] **마스터 노드는 EKS가 관리**한다는 사실을 근거로 사용했는가.
  - EKS에서는 컨트롤 플레인(API 서버, etcd, 스케줄러 등)을 AWS가 관리하므로, 클러스터를
    여러 개 만들어도 "마스터 노드 이중 구축/운영 부담"이 온프레미스만큼 크지 않다 —
    다만 클러스터당 별도 과금(EKS 클러스터 비용) + 워커 노드(EC2) 비용은 각각 발생한다는 점은
    여전히 비용 요인.
- [ ] **A(클러스터 분리) vs B(네임스페이스 분리)의 격리 수준 차이**를 구분했는가.
  - 네임스페이스는 논리적 분리일 뿐, 기본적으로 **RBAC/ResourceQuota/NetworkPolicy 없이는
    다른 네임스페이스의 리소스를 조작하는 것을 막지 못한다.** 이번 QA팀 사고(운영 Deployment를
    `scale 0`)는 네임스페이스만 나눴어도 **RBAC로 QA 계정의 kubectl 권한을 해당 네임스페이스로
    제한하지 않았다면 그대로 재발**할 수 있다는 점을 짚어야 한다.
  - 클러스터 분리(A)는 네트워크/노드/API 서버 수준까지 완전히 격리되므로 "실수로 다른 환경을
    건드릴 물리적 경로 자체가 없다"는 것이 핵심 장점.
- [ ] **비용/운영 복잡도** 트레이드오프를 언급했는가.
  - A: 클러스터 3개 → EKS 클러스터 비용 3배, 워커 노드/모니터링/CI-CD 파이프라인도 3세트 관리 필요.
  - B: 클러스터 1개 → 비용/운영 부담 적음. 대신 리소스 경합(같은 노드에서 dev 워크로드가 prod 워크로드의
    자원을 잠식) 가능성, 장애 전파 범위가 넓음(예: 클러스터 자체 장애 시 전 환경 영향).
- [ ] **결론적으로 이번 사고의 근본 원인이 "환경 분리 부재"가 아니라 "권한 분리(RBAC) 부재"일 수 있다는
  통찰**을 제시했는가 — 이게 나오면 가산점. (흔한 오답: "클러스터를 무조건 3개로 나눠야 한다"고만
  결론짓고 RBAC/권한 문제는 언급하지 않는 경우 — 부분 인정)

### 모범 답안 예시 (하나의 예시일 뿐)
> 초기에는 B(네임스페이스 분리 + RBAC로 QA 계정을 dev/staging 네임스페이스로 제한, ResourceQuota로
> 자원 상한 설정)로 시작하고, 조직/컴플라이언스 요구(예: 운영 DB 접근 로그 완전 분리, PCI-DSS 등)가
> 생기면 그때 prod만 별도 클러스터로 분리(A)하는 단계적 접근. 지금 사고의 직접 원인은 "환경 미분리"가
> 아니라 "QA 계정에 운영 리소스 조작 권한이 열려 있었다는 것"이므로 클러스터를 나누는 것만으로는
> 근본 해결이 안 된다.

## 문제 2 — 스케줄링 장애 진단 (핵심 포인트)

- [ ] **진단 순서**가 합리적인가. 기대하는 흐름:
  1. `kubectl get pods -o wide` (어떤 Pod가 Pending인지, 어느 노드에도 안 붙었는지 확인)
  2. `kubectl describe pod <pending-pod>` → `Events` 섹션에서 스케줄링 실패 사유 확인
     (예: `0/5 nodes are available: 5 Insufficient cpu` 같은 메시지가 결정적 단서)
  3. `kubectl get nodes` / `kubectl describe node <node>` → 노드별 Allocatable/Requests 확인
  4. (EKS라면) Cluster Autoscaler 동작 여부 확인 — 신규 워커 노드가 증설되고 있는지
     (`kubectl get pods -n kube-system | grep cluster-autoscaler`, ASG 상태 등)
- [ ] **직접 관련 구성요소로 kube-scheduler를 지목**했는가 — "새로 생성된 파드를 감지하고 실행할
  노드를 선택"하는 역할이므로, Pending은 스케줄러가 조건에 맞는 노드를 못 찾았다는 신호.
  kubelet은 "이미 노드에 배정된" 파드를 실행하는 역할이라 Pending 단계에는 아직 관여하지 않는다는
  구분까지 하면 가산점.
- [ ] **EKS에서 우리가 만질 수 있는 것 vs 없는 것 구분**:
  - 우리가 직접 볼 수 없음(관리형): kube-apiserver, etcd, kube-scheduler, kube-controller-manager
    자체의 로그/재시작 — AWS가 관리하는 컨트롤 플레인이라 접근 불가 (CloudWatch로 컨트롤 플레인
    로그를 "조회"하는 것은 가능하지만 직접 조작/재시작은 불가).
  - 우리가 직접 만질 수 있음: kubelet, kube-proxy는 워커 노드(EC2) 위에서 동작하므로 노드에 접속하거나
    노드 자체(EC2 인스턴스)를 교체/재시작하는 것은 가능.
- [ ] **원인 후보 2가지 이상**:
  - 워커 노드의 CPU/메모리 요청량(`resources.requests`) 대비 클러스터 전체 여유 자원 부족 →
    Cluster Autoscaler로 신규 노드 증설 필요 (증설 지연/실패 가능성 포함)
  - `nodeSelector`/`nodeAffinity`/`taint-toleration` 설정으로 특정 노드군에만 스케줄되도록
    제한되어 있는데 해당 노드군에 여유가 없는 경우
  - PodDisruptionBudget이나 topologySpreadConstraints 등으로 특정 조건(AZ 분산 등)을 만족하는
    노드가 부족한 경우
  - (흔한 오답) "네트워크 문제"라고만 답하는 경우 — Pending은 네트워크 문제가 아니라 스케줄링
    단계의 문제이므로 오답 처리, 다만 유도 질문으로 왜 아닌지 되물어볼 것.
