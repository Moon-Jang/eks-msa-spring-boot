⚠️ 스포일러 — 채점 루브릭 (정답이 아니라 평가 기준)

이 문제는 정답이 하나로 정해지지 않는 설계/의사결정형 문제다. 아래는 "반드시 언급되어야 할
핵심 포인트"와 "흔한 오답/누락"을 기준으로 채점하고, 모범 답안은 하나의 예시일 뿐 유일한 정답은 아니다.

## 1. VPC/서브넷 설계

**핵심 포인트 (언급 여부 체크)**
- 워커 노드가 퍼블릭 서브넷 + (기본적으로) 퍼블릭 IP 자동 할당 대상이 될 수 있다는 점 → 불필요하게
  인터넷에서 직접 도달 가능한 공격 표면이 늘어남을 지적해야 함
- 이상적으로는 워커 노드는 **프라이빗 서브넷**에 두고, 아웃바운드 인터넷(이미지 pull, 외부 API 등)은
  **NAT 게이트웨이**를 통해 나가야 한다는 방향 제시
- "최소 변경" 제약을 고려한 절충안 언급: 완전 재구축 대신 ①기존 Default VPC에 프라이빗 서브넷 +
  NAT Gateway를 추가로 만들고 신규 노드그룹만 프라이빗으로 옮기는 점진적 마이그레이션, 또는
  ②최소한 워커 노드 보안그룹에서 불필요한 인바운드(0.0.0.0/0 등)를 차단하는 완화책 정도는 나와야 함
- (가점) NAT Gateway는 AZ당 과금이 발생하므로 예산 제약과 상충될 수 있음을 언급하면 우수

**흔한 오답/누락**
- "VPC를 새로 만들어야 한다"고만 답하고 예산 제약(문제에서 명시)을 무시한 경우 → 요구사항 미반영으로 감점
- 퍼블릭 서브넷의 구체적 리스크(직접 도달 가능성)를 언급하지 않고 막연히 "보안에 안 좋다"고만 답한 경우

## 2. 보안그룹

**핵심 포인트**
- RDS 인바운드는 **0.0.0.0/0이 아니라, EKS 워커 노드의 보안그룹을 소스로 지정**해야 한다는 점
  (CIDR이 아니라 Security Group ID를 소스로 거는 것이 정석 — 노드 IP가 바뀌어도 규칙 유지됨)
- 포트는 DB 엔진 포트(PostgreSQL이면 5432, MySQL이면 3306)만 최소 개방
- (가점) 로컬 개발/CI에서 직접 RDS에 접근해야 하는 경우 bastion host나 VPN/Session Manager를 통한
  접근 방식을 제안하면 우수 (퍼블릭 액세스 허용을 계속 유지하는 답은 감점 요인)

**흔한 오답/누락**
- 강의 자료의 "3306 보안그룹 추가"를 그대로 답습해서 "0.0.0.0/0으로 열면 된다"는 식으로 답한 경우 →
  이건 명백한 감점 대상 (실제로 이 레포 강의 자료가 "db 퍼블릭엑세스 허용"이라고 되어 있는데, 이걸
  그대로 정답으로 베끼면 안 된다는 걸 인지하고 있는지가 이 문항의 핵심 변별 포인트)

## 3. IAM

**핵심 포인트**
- `withOIDC: false`면 IRSA(IAM Roles for Service Accounts)를 못 쓴다는 점을 인지해야 함 → 그 결과
  파드 단위로 세밀한 권한을 줄 수 없고, 노드 전체가 공유하는 노드 IAM 역할의 권한을 모든 파드가
  사실상 공유하게 되는 구조적 리스크를 지적해야 함
- 실무 정석: **OIDC 프로바이더 활성화 + IRSA**로, 특정 서비스어카운트에만 필요한 최소 권한(S3 특정
  버킷 read 등)을 매핑하는 방식이 정답 방향
- (가점) EKS Pod Identity(IRSA의 최신 대안, 이 레포에서 eksctl 출력 로그에도 "pod identity associations"
  언급이 나왔음)를 언급하면 최신 트렌드를 알고 있다는 가점 요소

**흔한 오답/누락**
- "역할에 정책을 더 추가하면 된다"는 식으로 답하고 IRSA/OIDC 자체를 언급하지 않은 경우 → 핵심 개념
  누락으로 감점
- 노드 역할과 파드 권한을 구분하지 못하고 뭉뚱그려 답한 경우

## 4. 외부 관리형 DB(Aiven) vs AWS RDS

**핵심 포인트**
- 인터넷 공개 엔드포인트 접속의 리스크: 네트워크 경로가 AWS VPC 내부로 폐쇄되지 않고 퍼블릭
  인터넷을 경유함 → TLS(`ssl=require`)로 암호화는 되지만, 소스 IP 기반 접근 제어가 VPC 보안그룹만큼
  세밀하지 않고, DDoS/무차별 대입 공격 표면이 RDS 대비 넓음
- RDS로 되돌리는 게 정당화되는 경우: 이미 AWS 생태계(EKS, VPC)에 있고 네트워크 레이턴시/보안을
  최우선으로 할 때, 또는 Multi-AZ/자동 백업 등 AWS 통합 운영 도구를 활용하고 싶을 때
- Aiven 같은 외부 관리형 서비스가 정당화되는 경우: 특정 DB 엔진 버전/확장 기능을 AWS RDS보다 빨리
  지원받고 싶거나, 멀티클라우드 전략, 혹은 스터디/PoC처럼 비용·설정 편의성이 우선일 때
  (이번 세션에서 실제로 그런 이유로 전환했다는 맥락과 자연스럽게 연결지어 답하면 우수)

**흔한 오답/누락**
- "무조건 RDS가 낫다" 또는 "무조건 외부 서비스가 낫다"처럼 트레이드오프 없이 단정하는 경우
- ssl=require 옵션이 있으니 안전하다고만 답하고, 네트워크 경로/접근 제어 관점의 차이를 언급하지 않은 경우

## 종합 채점 가이드

- 4개 항목 중 핵심 포인트를 2개 이상 놓치면 "재학습 필요"
- 모든 항목에서 핵심 포인트는 짚었으나 트레이드오프/근거가 얕으면 "통과, 심화 학습 권장"
- 예산 제약을 반영한 절충안, IRSA/OIDC, 보안그룹 소스 지정 방식까지 정확히 짚으면 "우수"
