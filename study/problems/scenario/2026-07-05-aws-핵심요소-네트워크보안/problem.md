# 시나리오: ordersystem 운영 환경 네트워크/보안 감사

## 배경

당신은 이 레포의 `2.ordersystem` 스프링 백엔드를 실제 서비스로 운영 준비 중인 팀의 인프라 담당자다.
현재 스터디용으로 아래와 같이 EKS 실습 환경을 구성해서 써왔다.

- `eksctl`로 클러스터 생성 시 **Default VPC를 그대로 재사용** (`cluster.yaml`에 `vpc.id`로 지정),
  새 VPC를 만들지 않았다.
- Default VPC의 서브넷 4개(`ap-northeast-2a~d`)가 전부 **퍼블릭 서브넷**이고, 워커 노드(`t3.small` 2대)가
  이 퍼블릭 서브넷 위에 떠 있다.
- 클러스터 IAM 설정은 `iam.withOIDC: false`이고, 노드 IAM 역할(`AmazonEKSNodeRole`)은 콘솔에서
  "권장 역할"로 생성한 것을 그대로 쓰고 있다 (역할에 어떤 정책이 붙어있는지 세부 검토는 안 한 상태).
- DB는 원래 RDS MySQL을 쓰다가, 이번에 외부 관리형 서비스(Aiven Postgres)로 전환했다.
  `application-prod.yml`에서 `jdbc:postgresql://${DB_HOST}:15210/defaultdb?ssl=require`로 접속하며,
  이 DB는 인터넷에 공개된 엔드포인트를 통해 접근한다 (AWS VPC 내부 리소스가 아님).
- 팀 리드가 "이제 실습은 끝났으니, 이 구성 그대로 실제 프로덕션에 올려도 되는지 검토해서 보고해달라"고
  요청했다. 단, 예산 문제로 이번 분기에 VPC를 완전히 새로 설계할 여력은 없고, **현재 리소스를 최소
  변경으로 개선**하는 방향을 우선 검토해야 한다.

## 요구사항

다음 4가지 항목에 대해, 각각 **문제점 진단 → 개선 방향과 근거 → (가능하면) 트레이드오프**를 서술하라.

1. **VPC/서브넷 설계**: 워커 노드가 퍼블릭 서브넷에 떠 있는 현재 구성의 문제점은 무엇이고,
   최소 변경으로 개선하려면 어떻게 할 것인가? (완전 재구축 대비 절충안 포함)
2. **보안그룹**: 만약 DB를 다시 AWS RDS로 되돌린다고 가정할 때, RDS의 인바운드 보안그룹 규칙을
   어떻게 설계해야 하는가? (현재처럼 "3306 보안그룹 추가" 수준이 아니라, 구체적으로 어떤 소스를
   허용해야 하는지)
3. **IAM**: 현재 `iam.withOIDC: false` + 콘솔 권장 역할 그대로 쓰는 구성의 리스크는 무엇이며,
   실무에서는 EKS 파드가 AWS 리소스(S3, RDS 등)에 접근해야 할 때 어떤 방식을 쓰는 게 정석인가?
4. **외부 관리형 DB(Aiven) vs AWS RDS**: 지금처럼 인터넷 공개 엔드포인트로 외부 DB에 접속하는 구성이
   가진 위험은 무엇이며, RDS로 되돌리는 것과 비교했을 때 각각 언제 정당화될 수 있는가?

형식은 자유 서술(마크다운)이며, 필요하면 보안그룹 규칙 예시나 IAM 정책 스니펫을 포함해도 된다.
