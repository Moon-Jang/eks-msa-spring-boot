# 채점 결과 — max (시나리오 ② Deployment 배포 전략)

> 채점일: 2026-07-09 · 대상: `answer-max.md` · 기준: `answer-key.md` 루브릭
> 서술형 시나리오이므로 클러스터 검증 없이 정적 평가.

## 종합 판정: 부분 통과

핵심 명령어와 (a)/(b) 판정은 대부분 맞히고 "흔한 오답"을 회피했으나, 개념 서술 2곳이 어긋나고
근거가 얇다("모름" 2건 포함).

---

## (1) 왜 롤링 업데이트인가 — 부분 충족

**제출 요지**
> "롤링은 배포 중 새 Pod에 문제가 있어도 아직 안 바뀐 기존 Pod가 요청을 받아준다. Recreate(전부 내리기)는
> 배포되는 Pod에 문제가 있을 시 서비스가 중단된다."

**맞은 점**
- "순차 교체 중에도 아직 안 바뀐 기존 정상 Pod가 요청을 받는다" → "배포 중에도 일부 Pod가 살아 있다"는
  루브릭 핵심에 근접.

**틀린 점 ① — 다운타임 감소 메커니즘 오해**
- 제출은 "새 Pod에 **문제가 있어도** 기존 Pod가 받아준다"로 프레이밍 → 롤링을 사실상
  **카나리/장애격리**처럼 이해한 것.
- 루브릭 핵심은 **새 Pod의 결함 여부와 무관**하다: 롤링은 배포 내내 **최소 처리 용량(살아있는 Pod 수)이
  0으로 떨어지지 않도록** 신 Pod를 먼저 띄운 뒤 구 Pod를 순차 종료하는 것이다. "장애 대비"가 아니라
  "가용 용량 유지"가 본질.

**틀린 점 ② — Recreate 다운타임 원인 오설명**
- 제출: "Recreate는 배포되는 Pod에 **문제가 있을 시** 서비스 중단."
- 부정확하다. Recreate는 새 Pod가 **완벽히 정상이어도** 기존 Pod를 전부 내린 시점부터 새 Pod가
  Ready 될 때까지 **Pod 수가 0인 구간**이 생겨 그 사이 5xx가 난다. "문제가 있을 때만" 중단되는 게 아니다.
- 무중단이 최우선인 이 시나리오에서 다운타임의 원인을 잘못 짚은 것이라 실무적으로 의미 있는 오해.

**놓친 점 (범위 밖, 감점 없음)**
- `maxUnavailable/maxSurge` 미언급. 문제에 "노드 리소스 여유 적음" 제약이 있었으므로
  `maxSurge:1 / maxUnavailable:0`을 냈으면 가점 구간이었다.

## (2) 리비전과 재배포 — 부분 충족

**맞은 점**
- (a) "바뀐다" ✅ / (b) "안 바뀐다" ✅ → 루브릭이 지목한 흔한 오답("replicas 변경으로 새 revision")을 회피.
- latest 강제 재배포에 `kubectl rollout restart`를 정확히 지목 → "delete로 지운다"류 오답 회피.

**틀린/놓친 점 ① — (a)/(b) 근거를 "모름"으로 비움**
- 판정만 있고 이유를 적지 않았다. 문제와 루브릭이 요구한 근거는:
  - (a) 이미지 태그는 `spec.template` 안의 값 → template이 바뀌므로 새 ReplicaSet(revision) 생성.
  - (b) `replicas`는 `spec.template` **바깥** 필드 → template 불변 → revision 무관, 파드 수만 조정.
- 5년차 기준으로는 "정답은 맞췄지만 왜인지 설명 못 함"이 가장 아쉬운 지점. 판정 근거를 언어화할 것.

**틀린/놓친 점 ② — restart 메커니즘**
- 제출: restart의 효과를 "컨테이너 이미지를 새로 받아서 시작하기 때문"으로만 설명.
- 문제는 "그 명령이 **왜** 그런 효과를 내는지"를 물었다. 실제 메커니즘:
  restart는 pod template에 **`restartedAt` 어노테이션을 주입** → template 해시 변경 →
  **새 ReplicaSet(=새 revision) 생성** → 파드 재생성. 이미지 재수신은 `imagePullPolicy: Always`가
  전제된 **부수 효과**일 뿐이다.
- 즉 본질("새 revision이 생긴다")을 빠뜨리고 부수 효과("이미지를 받는다")만 설명했다.

## (3) 롤백 — 부분 충족

**맞은 점**
- `kubectl rollout undo deployment <name>` 명령 정확 ✅ (문법 유효).
- 내부 동작을 "이전 ReplicaSet이 복구되면서 Pod들이 되살아난다"로 서술 → 단순 이미지 재apply가 아니라
  **RS 전환**이라는 루브릭 핵심을 짚음(가점).
- (보완하면 완전: 이전 RS를 scale-up 하면서 **현재 RS를 scale-down**한다는 반대편까지 언급.)

**틀린 점 — latest 태그 문제 논지 어긋남 (핵심 필수 항목)**
- 제출: "latest는 새로운 컨테이너를 받지 않아서 롤백할 곳을 찾지 못한다." → 뜻이 어긋난다.
- 루브릭 핵심: latest는 **가변 태그**라 각 revision이 어떤 이미지였는지 특정할 수 없고, `undo`로 이전
  RS를 되살려도 그 RS 역시 latest를 가리켜 **같은 (버그 있는) 이미지를 다시 pull → 버그 재현**된다.
  그래서 **불변 태그(커밋 SHA)**를 써야 한다. 제출 문장은 이 논지를 담지 못해 사실상 미충족.

**범위 밖 (DB 스키마)**
- "기존 DB를 죽이고 새로 만드는 게 아니라 DB는 롤백 안 됨" → DB가 코드처럼 되돌아가지 않는다는 인식은
  있음(소폭 가점). 다만 루브릭의 구코드-신스키마 불호환 + expand/contract 단계 분리까지는 못 감
  (범위 밖이라 감점 없음).

## kubectl 문법 검증

| 제출 표현 | 판정 | 비고 |
|---|---|---|
| `kubectl rollout undo deployment name` | 유효 | `--to-revision=N`을 붙였으면 예측 가능성 가점이었음 |
| `kubectl rollout restart` (리소스 인자 없음) | 불완전 | 대상 없이는 실행 불가 → `kubectl rollout restart deployment <name>` |

## 복습 포인트
1. 롤링의 본질은 "장애 대비"가 아니라 **가용 용량이 0으로 안 떨어지게 유지**하는 것. Recreate 다운타임은
   Pod 결함과 무관하게 "Pod 수 0 구간" 때문에 발생.
2. (a)/(b) 판정 근거를 언어화: `spec.template` 안/밖 필드 구분.
3. `rollout restart`의 본질은 새 revision 생성(어노테이션 주입), 이미지 재pull은 부수 효과.
4. latest 롤백 함정: 같은 이미지 재pull로 버그 재현 → 불변 태그(SHA) 사용.
