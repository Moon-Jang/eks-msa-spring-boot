# k8s 실무 시나리오 문제 출제 스킬

이 레포는 인프런 "EKS를 활용한 Spring 운영서버 배포" 강의 실습 레포이며, 스터디원들은 5년차 이상
현업 개발자다. **자격증 덤프형(단답/암기) 문제는 지양**하고, 강의에서 다룬 개념을 실제 운영 환경
의사결정/트러블슈팅에 적용하는 문제를 만든다.

## 문제 유형 (섞어서 출제)

1. **아키텍처 의사결정형**: 트래픽 패턴/조직 제약을 제시하고 근거 있는 선택을 요구
   (예: "피크 시간대 트래픽이 급증하는 주문 서비스에 Rolling Update vs Canary, 어느 쪽을 쓰고
   `maxUnavailable`/`maxSurge`는 어떻게 잡을지 근거와 함께 서술하라")
2. **핸즈온 트러블슈팅형**: 실제로 `kubectl` 명령을 실행해 문제를 해결하도록 요구
   (예: "아래 상황에서 Ingress 통해 503이 발생한다. 원인을 특정하고 고쳐라" — 단, k8s-villain-mode의
   "버그 주입" 방식과 달리 반드시 매니페스트를 조작할 필요는 없고, 설계/설정 누락형 문제도 가능)
3. **마이그레이션/설계형**: 레거시 인프라를 이 레포의 강의 범위(EKS, Ingress, HPA, CI/CD, Secret 등)로
   전환한다고 가정하고 설계를 요구 (예: "온프레미스 VM 기반 서비스를 EKS로 옮긴다면 네트워킹/Secret
   관리/무중단 배포를 어떻게 재구성할지")

CKA 실기 문제 형식(제한시간 내 리소스 생성/수정)을 참고하되, 실무 맥락 없는 단순 명령어 암기
문제(예: "파드를 생성하는 명령어를 써라")는 만들지 않는다.

## 실행 절차

1. **범위 확인 (필수 질문)**: 사용자가 이미 구체적인 주제를 명시하지 않았다면, 문제를 작성하기 전에
   반드시 "지금 어느 부분을 공부하고 있나요?"라고 먼저 물어본다 (`contents/` 목차를 옵션으로 참고 제시
   가능). 임의로 최신 파일을 추측해 진행하지 않고, 사용자의 답을 받은 뒤 2단계로 진행한다.
2. **문제 작성**: 위 유형 중 1~2개를 조합해 하나의 시나리오를 만든다. 반드시 다음을 포함한다.
   - 배경 상황(회사/서비스 맥락, 현재 레포의 강의 시나리오와 자연스럽게 이어지는 설정 — 예: 이 레포의
     `4.ordersystem` 스프링 백엔드를 운영 중이라는 가정)
   - 요구사항 (설계 결정 + 근거, 또는 실제로 적용해볼 명령/매니페스트)
   - 제약 조건 (트래픽/가용성/비용/조직 정책 등 현실적인 트레이드오프가 있어야 함)
3. **채점 기준 작성**: 정답이 하나로 정해지지 않는 문제가 많으므로, `answer-key.md`는 "정답"이 아니라
   **평가 루브릭**으로 작성한다 — 반드시 언급해야 할 핵심 포인트 목록, 흔한 오답/누락 포인트,
   모범 답안 예시(하나의 예시일 뿐 유일해는 아님을 명시).
4. **결과물 생성**: `study/problems/scenario/<YYYY-MM-DD>-<주제-슬러그>/` 에 아래 파일 생성.
   - `problem.md`: 풀이자에게 전달할 시나리오 + 요구사항
   - `answer-key.md`: 채점 루브릭 (파일 맨 위에 `⚠️ 스포일러` 명시). `.gitignore`로 커밋 제외됨.
5. 사용자에게는 문제 요약만 보여주고 채점 기준의 세부 내용은 노출하지 않는다.

## 정답 확인

풀이자의 답변(서술형 또는 명령/매니페스트)이 나오면 `k8s-answer-checker` 에이전트에게
문제 폴더 경로와 제출 내용을 전달해 루브릭 기반 채점을 위임한다.
