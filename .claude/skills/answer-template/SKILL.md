# k8s 스터디 답안 템플릿 생성 스킬

문제 파일(`problem.md`)의 요구사항을 **항목 전문을 그대로 복붙하지 않고 한 줄로 요약**한 뒤, 각 항목
바로 아래에 **빈 정답란**을 둔 답안 템플릿을 만든다. 풀이자는 요약만 보고 무엇을 묻는지 파악한 채로
정답란만 채우면 된다.

- 이 스킬은 **빈 템플릿 생성**까지만 한다. 답 내용은 절대 지어내지 않는다(내용은 풀이자 소유).
- 작성된 답안을 커밋·PR 하는 것은 별도 스킬 [[answer-submit]]이 담당한다. (파일명/브랜치 컨벤션 준수)

## 출력 파일 컨벤션

- 위치: 대상 문제 폴더 안 — `study/problems/scenario/<YYYY-MM-DD>-<주제-슬러그>/`
- 파일명: **`answer-<핸들>.md`** (예: `answer-dave.md`)
  - `<핸들>`은 풀이자의 스터디 핸들. 이후 `answer-submit`의 브랜치/PR 핸들과 동일하게 맞춘다.
- ⚠ 예약 파일명 금지: **`answer-key.md`** 는 채점 루브릭 전용(gitignore 제외 대상)이므로 겹치지 않게 한다.

## 요약 원칙 (핵심)

문제의 각 요구사항 항목을 다음 규칙으로 압축한다.

1. **묻는 것만 남긴다.** 배경 설명·부연·힌트는 걷어내고 "무엇을 답해야 하는지"만 1~2줄로.
2. **챕터 핵심 개념 키워드는 보존한다.** (예: `namespace`, `Service 디스커버리`, `롤링 업데이트`,
   `ReplicaSet/revision`, `Ingress`) — 풀이자가 어느 개념으로 답해야 하는지 힌트가 되도록.
3. **`⚠ 강의 범위 밖` 표시는 반드시 유지한다.** 요약에서도 `(+ ⚠ 범위 밖: … / 몰라도 됨)` 형태로
   붙여, 필수와 가점(선택)을 구분해준다. 이 구분을 지우면 안 된다.
4. 한 항목 안에 하위 질문이 여러 개면 `·`로 이어 한 줄에 담거나, 필요한 만큼 하위 불릿으로 나눈다.
5. 원문 표현을 창작으로 바꾸지 말 것 — 요약이지 재해석이 아니다. 애매하면 원문 키워드를 그대로 쓴다.

## 정답란 형식

각 항목 요약 바로 아래에 아래 형태의 빈 정답란을 둔다.

```
**답:**

_(작성)_
```

- `_(작성)_` 자리표시자는 풀이자가 지우고 답을 채운다.
- 항목 사이는 `---` 구분선으로 나눠 가독성을 높인다.

## 템플릿 골격

```markdown
# 답안: <문제 제목>

> 문제: <problem.md 상대경로>
> 출제 범위: <problem.md의 출제 범위 한 줄>
> 풀이자: <핸들>

---

## (1) <항목 소제목>
**묻는 것:** <한 줄 요약> (+ ⚠ 범위 밖: <있으면> / 몰라도 됨)

**답:**

_(작성)_

---

## (2) <항목 소제목>
...
```

## 실행 절차

1. **입력 확인 (필수)**: 아래 두 값을 반드시 확보한다. 하나라도 없으면 **템플릿을 만들기 전에 물어본다.**
   - (a) 대상 문제 폴더(또는 `problem.md` 경로)
   - (b) **작성자(풀이자 핸들)** — 파일명 `answer-<핸들>.md`와 템플릿 `> 풀이자:` 줄에 그대로 들어간다.
     이후 [[answer-submit]]의 브랜치/PR 핸들과 동일해야 하므로 임의로 정하지 말고 반드시 입력받는다.
2. **problem.md 정독**: 헤더(제목/출제 범위/형식), `## 요구사항`의 각 항목 `(1)(2)…`, 하위 불릿,
   `⚠ 강의 범위 밖` 표시, `## 제약 조건`을 모두 읽는다.
3. **요약 생성**: 위 "요약 원칙"대로 각 요구사항 항목을 압축한다. `## 제약 조건`은 답에 영향을 주므로
   템플릿 하단에 `> 제약(참고):` 한 줄 요약으로 남긴다(정답란은 안 붙임).
4. **파일 작성**: `answer-<핸들>.md`를 문제 폴더에 생성한다. 이미 있으면 **덮어쓰기 전에 사용자에게
   확인**한다(작성 중인 답이 날아갈 수 있음).
5. **결과 보고**: 생성 경로와 항목 수를 알린다. 이어서 답 작성 후 [[answer-submit]]으로 제출하면
   된다고 안내한다.

## 참고 예시

`study/problems/scenario/2026-07-09-full-stack-deploy-flow/problem.md` 의 (1) 항목을 요약하면:

- 원문: "dev와 운영을 어떻게 나눌지 결정하고, 챕터에서 배운 namespace로 어떻게 분리할지 설계하라."
  + "`⚠ 강의 범위 밖`: namespace만으로는 막지 못하는 것(권한/네트워크 격리) …"
- 요약: **묻는 것:** dev/운영을 `namespace`로 어떻게 분리 설계할지. (+ ⚠ 범위 밖: namespace만으론 못
  막는 것(권한·네트워크 격리) / 몰라도 됨)
